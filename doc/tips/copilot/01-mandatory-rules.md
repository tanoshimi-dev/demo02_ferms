# Copilot 必須ルール

このファイルは、`demo02_ferms` で Copilot が**必ず守るべきルール**をまとめたものです。  
作業前に必ず参照させます。

## 1. 基本原則

1. 仕様を読まずに実装を始めない
2. 不明点がある場合は、推測で広げすぎず、既存文書の範囲で判断する
3. 既存方針に反する独自設計を勝手に持ち込まない
4. 実装対象に直接関係ない箇所を勝手に直さない
5. 一時的な対症療法ではなく、仕様と整合する形で直す

## 2. 技術選定ルール

1. フロントエンドは **Nuxt + Vue 3 + TypeScript**
2. バックエンドは **NestJS + TypeScript**
3. データベースは **PostgreSQL**
4. 認証は **portal 認証引き継ぎ + アプリ側ローカルセッション**
5. 実装は `doc\policy.md` の方針を優先する

## 3. 認証ルール

1. 独自ログイン画面を新設しない
2. `portal_token` をフロントエンド JavaScript で直接扱わない
3. 認証の最終判定はバックエンドで行う
4. 未認証時は portal 側ログイン導線へ戻す前提を崩さない
5. `demo01_crm` と整合しない認証方式へ変更しない

## 4. 実装ルール

1. Nuxt の規約ベース構成を崩しすぎない
2. NestJS は feature module 単位で責務分離する
3. 入力検証は DTO class と `ValidationPipe` を前提にする
4. 予約可否、重複予約防止、権限制御はバックエンドで保証する
5. 共有ロジックは composable や service に寄せ、重複実装を避ける

## 5. ドキュメントルール

1. 仕様変更を伴う実装では、関連文書も必要に応じて更新する
2. 実装判断は README ではなく `doc\spec` と `doc\dev-plan` を正本とする
3. 認証関連は `doc\spec-auth\README.md` を正本とする
4. 毎回の temporary note を勝手に増やさず、恒久価値のある情報だけ文書化する

## 6. 変更禁止・注意事項

1. 本番構成や本番環境依存の内容を、指示なしで READMEや doc に書かない
2. mock 認証を本番前提の説明に混ぜない
3. セキュリティ上重要な設定値をソースに直書きしない
4. 不要なライブラリを増やさない
5. 未依頼の大規模リファクタリングをしない

## 7. 作業前チェック

作業開始前に最低限、以下を確認する。

- 今回のタスクに関係する仕様書はどれか
- 認証に影響するか
- 予約ルールに影響するか
- 管理画面と利用者画面の両方に影響するか
- ドキュメント更新が必要か
